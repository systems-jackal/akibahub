# OWASP Top 10 (2021) Mapping

*Last updated: 2026-07-14, following the security hardening pass documented in `CHANGELOG.md` [0.2.0]. Previous version of this document is preserved in git history for comparison — several items marked as gaps there are now closed.*

| # | Risk | How Akiba Hub Addresses It | Status / Notes |
|---|------|----------------------------|----------------|
| A01 | **Broken Access Control** | Spring Security + JWT; every group-scoped read/write now enforces membership via a shared `requireMembership()` check (previously `getGroup`, `getGroupMembers`, and group/proposal listing endpoints had no such check — an IDOR, fixed in 0.2.0). | No formal role-based access control (RBAC) yet — all authenticated users have equivalent privilege within their own group memberships. Acceptable for current feature set; would need real roles before adding admin/support tooling. |
| A02 | **Cryptographic Failures** | Passwords hashed with BCrypt; JWT signed with HMAC‑SHA256; refresh tokens stored as SHA‑256 hashes, never in plaintext. | **HTTPS enforcement still depends on the Nginx layer outside this repo** — not verifiable from application code alone. JWT secret still has a weak literal default checked into `application.yml` (should be removed entirely so the app fails to start without an explicit `JWT_SECRET`). |
| A03 | **Injection** | JPA parameterized queries throughout; no raw/native SQL in application code. | Unchanged from MVP — still a strength. |
| A04 | **Insecure Design** | Threat modeling now documented (`threat_model.md`); double-entry ledger introduced specifically to make the negative-amount exploit structurally impossible rather than relying only on input validation; idempotency keys prevent double-submit-driven design flaws. | M-Pesa integration is still design-only (`docs/architecture/payhero-integration.md`) — its actual security properties (webhook signature verification, callback idempotency) can't be assessed until implemented. |
| A05 | **Security Misconfiguration** | CORS restricted to a specific production origin; stack traces disabled in error responses; `open-in-view` disabled; `ddl-auto` changed from `update` to `validate` with Flyway owning schema changes. | Weak default secrets still committed to `application.yml` (see A02). Database port still published to the host in `docker-compose.prod.yml` — unnecessary attack surface. |
| A06 | **Vulnerable and Outdated Components** | Spring Boot 3.3.0, recent Flyway/MariaDB versions. `dependency-check-maven` is declared in `pom.xml`. | **The dependency-check plugin is declared but never actually invoked in CI** (`deploy.yml` only runs `mvn package`) — this is a false sense of coverage and should be treated as still open. |
| A07 | **Identification and Authentication Failures** | Rate limiting fixed (previously bypassable via a spoofed `X-Forwarded-For` header — closed in 0.2.0); access tokens now short-lived (15 min, down from 1 hour); refresh tokens are individually revocable, rotated on use, and revoked on password change/logout — the "no token revocation" gap from the original threat model is now closed for refresh tokens (an already-issued 15-minute access token still can't be revoked early, but the exposure window is now minutes, not an hour+). | Still no MFA/step-up authentication for high-value actions (large withdrawals, password/phone changes) — flagged as a market-readiness gap, not yet built. Still no account lockout after repeated failed logins beyond the rate limiter. |
| A08 | **Software and Data Integrity Failures** | CI/CD pipeline now exists (GitHub Actions → GHCR); Flyway migrations are version-controlled, giving an auditable history of every schema change. | CI still doesn't run the declared dependency-check step (see A06) or any container image scan (e.g. Trivy). Deploy pipeline uses a floating `:latest` tag rather than a pinned, reproducible SHA-tagged image. |
| A09 | **Security Logging and Monitoring Failures** | Audit logging remains comprehensive; unexpected server errors are now logged with a correlated short reference ID instead of being silently swallowed or leaked to the client (`GlobalExceptionHandler`). | Still no centralized log aggregation, metrics, or alerting — an anomaly (e.g. a ledger imbalance, or a spike in failed logins) would currently be discovered by a user complaint, not a system alert. This remains the most significant monitoring gap. |
| A10 | **Server‑Side Request Forgery (SSRF)** | No URLs are fetched from user input; low risk in the current feature set. | Unchanged — will need explicit revisiting once M-Pesa/Payhero webhook handling is actually implemented, since that introduces an inbound callback surface that needs signature verification, not an SSRF vector per se, but adjacent risk worth re-assessing at that point. |
